# pangreksa-service

The shared **domain core** of Pangreksa HRIS. Packaged as a plain Maven jar (no Spring Boot main class, no `spring-boot-starter` that triggers auto-config). Both `pangreksa-be` and `pangreksa-web` depend on this artifact.

## Maven coordinates

```xml
<dependency>
    <groupId>com.pangreksa</groupId>
    <artifactId>pangreksa-service</artifactId>
    <version>1.0.0-SNAPSHOT</version>
</dependency>
```

## Hard rules

- **NO Spring Web.** No `org.springframework.web.*` imports. No `@RestController`, `@RequestMapping`, `ResponseEntity`.
- **NO Vaadin.** No `com.vaadin.*` imports.
- **NO Spring Security web layer.** No `SecurityFilterChain`, no login views. (Method-level `@PreAuthorize` is OK.)
- **NO main class, NO `@SpringBootApplication`.** This is a library.
- **NO transport DTOs.** Request/response shapes live in adapter modules. Use-cases accept and return domain types or use-case-specific records defined alongside the use case.
- **`@Entity` only here.** If you find yourself adding `@Entity` in BE or Web, stop ‚Äî it belongs here.
- **Repositories are interfaces only** (`port.out`). Implementations are auto-generated by Spring Data in the consumer apps when they `@EnableJpaRepositories(basePackages = "com.pangreksa.service.port.out")`.

## Conventions

- **Entity naming:** `Hr*` for HR domain (`HrPerson`, `HrLeaveApplication`), `Fw*` for framework/auth (`FwAppUser`, `FwResponsibilities`).
- **Auditable rows** extend `AuditableEntity<ID>` (adds `createdBy`, `updatedBy`, `createdAt`, `updatedAt`).
- **IDs** are generated from sequences `global_hr_seq` / `global_fw_seq` (`@GeneratedValue(strategy = SEQUENCE, generator = "...")`).
- **Lombok:** `@Getter` `@Setter` `@NoArgsConstructor` `@AllArgsConstructor` `@Builder` on entities. **Do NOT use `@Data`** on `@Entity` classes (it generates equals/hashCode from all fields, which breaks Hibernate proxies and lazy collections). Use `@Getter` + `@Setter` + the equals/hashCode from `AbstractEntity` (id-based).
- **Use cases:** one interface in `port.in`, one impl in `application.usecase`. Impl is `@Service`, methods `@Transactional` (read-write) or `@Transactional(readOnly = true)`. Constructor injection only (`@RequiredArgsConstructor`).
- **Repositories:** prefer `@EntityGraph` to avoid N+1; use `@Query` JPQL for anything beyond derived names.

## Build & test

```bash
./mvnw clean install
```

Tests live in `src/test/java/com/pangreksa/service/`:
- `domain/` ‚Äî pure unit tests for entity behavior and enum logic.
- `application/` ‚Äî use-case tests with in-memory fakes implementing `port.out` interfaces (no Spring context, no DB).
- `architecture/` ‚Äî ArchUnit rules enforcing the "no Vaadin / no Spring Web / no security web" bans listed above.

A Testcontainers-backed integration test layer can live here too, but most integration testing happens in the consumer apps where the full Spring context exists.

## When in doubt

Ask: *"Is this transport-specific (HTTP / Vaadin UI / Flutter) or domain-specific (rules about leave, attendance, payroll)?"* If transport, it belongs in the adapter. If domain, it belongs here.
